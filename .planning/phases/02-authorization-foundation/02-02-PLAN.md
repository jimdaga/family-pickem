# Phase 02 Plan 02: View/API Guards And Proof Integration

## Goal

Adapt the core authorization helpers to Django page views and DRF/API views, then prove the behavior through a small integration surface.

## Scope

- Add browser/page guard decorators that map helper denials to login redirect, 404, and 403.
- Add DRF/API guard or permission adapter that maps helper denials to API-safe 401/403/404 responses.
- Add a tiny proof endpoint or test-only route demonstrating family/pool membership enforcement.
- Add integration tests for anonymous, non-member, wrong-role, active member, admin, owner, and family/pool mismatch behavior.

## Out of Scope

- Migrating dashboard, picks, standings, scores, message board, profile, or commissioner pages.
- Adding family switcher/onboarding URLs.
- Replacing legacy global commissioner pages.

## Files Likely To Change

- `pickem/pickem_homepage/authz.py` (new) or equivalent decorator module
- `pickem/pickem_api/permissions.py`
- `pickem/pickem_api/views.py` if adding API proof endpoint
- `pickem/pickem_api/urls.py` if adding API proof endpoint
- `pickem/pickem_homepage/tests.py` and/or `pickem/pickem_api/tests.py`

## Tasks

1. Add tests for browser decorator behavior using a small guarded test/proof view.
2. Add tests for API adapter behavior using a small guarded test/proof endpoint.
3. Implement browser decorators around `pickem_api.authz` helpers.
4. Implement DRF/API adapter around `pickem_api.authz` helpers.
5. Add the smallest useful proof route only if it does not broaden public surface or leak tenant existence.
6. Run focused homepage/API tests and commit atomically.

## Security Considerations

- Anonymous browser/page requests redirect to login.
- Anonymous API requests return an auth error without pretending the tenant exists.
- Authenticated non-members receive 404.
- Active members with insufficient role receive 403.
- Do not put role restrictions in hidden UI only; all checks stay server-side.
- Keep response bodies generic for denied tenant access.
- Do not add `csrf_exempt` to new proof POST routes; prefer GET-only proof or CSRF-protected POST if needed.

## Tests To Add/Update

- Browser proof: anonymous redirects.
- Browser proof: outsider receives 404.
- Browser proof: active member allowed for member route.
- Browser proof: member receives 403 for admin route.
- Browser proof: admin/owner allowed for admin route.
- API proof: outsider receives 404.
- API proof: wrong role receives 403.
- API proof: pool-family mismatch receives 404.

## Manual QA Checklist

- Start local server if needed.
- Hit allowed proof route as a logged-in test user if local auth setup supports it.
- Confirm denied proof route does not expose inaccessible family/pool names.

## Rollback Considerations

If the proof route is undesirable, revert only the route/view and keep helper/decorator tests where possible. Downstream phases should depend on the helpers, not on the proof route.

## Definition Of Done

- View/API adapters exist and share the core helper semantics.
- Integration tests prove the locked 401/redirect, 404, and 403 behavior.
- No broad product routes have been migrated prematurely.
